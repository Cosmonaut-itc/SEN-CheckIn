# Release 02 – Integración con Amazon Rekognition (enrolamiento + identificación)

## Estado actual de la implementación

### ✅ Ya implementado

- **Paquete `@sen-checkin/api-contract`**: Existe y tiene configuración base de Eden Treaty
    - Cliente configurado (`createApiClient`, `apiClient`)
    - Tipo `AppType` exportado
    - Listo para extender con nuevos endpoints

### ❌ Pendiente de implementar

- **Paquete `@sen-checkin/types`**: Existe pero está vacío (solo placeholder)
- **Servicio de Rekognition**: No existe (`apps/api/src/services/` no existe)
- **Endpoints de enrolamiento e identificación**: No implementados
- **Esquema de base de datos**: Solo tiene tablas de autenticación (user, session, account, verification)
    - Falta tabla/modelo para empleados (employees)
    - Falta tabla opcional para registro de enrolamientos faciales
- **Variables de entorno AWS**: No documentadas ni configuradas
- **Dependencias AWS SDK**: No instaladas en `apps/api/package.json`

---

## Rol de esta release

Añadir la capa de biometría facial:

- Integrar Amazon Rekognition desde el backend.
- Definir el modelo conceptual de enrolamiento de rostro por empleado.
- Exponer endpoints para:
    - Enrolar la cara de un empleado.
    - Identificar a un empleado a partir de una imagen.
- Mantener type safety end-to-end a través de `@sen-checkin/types` y `@sen-checkin/api-contract`.

No se define aún la lógica final de check-in/check-out, solo la capacidad de identificar.

---

## Objetivos concretos

1. Configurar el cliente de Rekognition en `apps/api`.
2. Definir funciones internas de servicio para:
    - Enrolar rostro de un empleado.
    - Buscar empleados por imagen.
3. Definir tipos para resultados de reconocimiento en `@sen-checkin/types`.
4. Exponer endpoints API y extender contrato Eden (el contrato base ya existe).

---

## Paso 1 – Configuración de Rekognition

**Estado**: Pendiente. No hay configuración de AWS ni dependencias instaladas.

### 1.1 Variables de entorno

Definir y documentar variables de entorno necesarias:

- `AWS_REGION` (región de Rekognition).
- `AWS_ACCESS_KEY_ID`.
- `AWS_SECRET_ACCESS_KEY`.
- `AWS_REKOGNITION_COLLECTION_ID` (ID de la colección donde se guardan las caras).

Indicar en comentarios:

- La colección puede crearse manualmente en AWS o vía script aparte.
- Para simplificar, se asume que la colección ya existe antes de usar los endpoints.

### 1.2 Cliente de Rekognition

**Estado**: Pendiente. El directorio `apps/api/src/services/` no existe.

En `apps/api/src`:

- Crear el directorio `services/` si no existe.
- Definir un módulo de servicio, p.ej. `services/face-recognition.ts`.
- Instalar dependencia: `@aws-sdk/client-rekognition` en `apps/api/package.json`.
- Dentro, en pseudocódigo:
    - Crear un cliente de Rekognition usando el SDK v3 de AWS.
    - Definir funciones internas:
        - `enrollFace(employeeId, imageBytes)`:
            - Llamar a la operación equivalente a “IndexFaces”.
            - Usar el `employeeId` como `ExternalImageId` para que luego se pueda mapear el rostro al empleado.
        - `searchFace(imageBytes)`:
            - Llamar a la operación equivalente a “SearchFacesByImage”.
            - Devolver la coincidencia más relevante (face match con mayor similitud).

Indicar que las funciones devuelven estructuras de datos simplificadas, no el objeto completo del SDK.

## Paso 2 – Tipos de reconocimiento en `@sen-checkin/types`

**Estado**: Pendiente. El paquete `@sen-checkin/types` existe pero está vacío (solo contiene un placeholder).

En `packages/types`:

- Añadir tipos relacionados con biometría:
    - `FaceEnrollmentResult`:
        - Identificador de la cara en Rekognition (FaceId).
        - `employeeId` asociado.
        - Metadatos básicos (fecha de enrolamiento, etc.), si se desea.
    - `FaceMatch`:
        - `employeeId` o referencia al empleado.
        - Puntaje de similitud (porcentaje).
    - `RecognitionResult`:
        - Flag `matched` (booleano).
        - `match` opcional (`FaceMatch`).

Estos tipos se usarán tanto en el backend como en la app Expo para interpretar resultados.

## Paso 3 – Endpoints Elysia para enrolamiento e identificación

**Estado**: Pendiente. El servidor Elysia solo tiene endpoints de autenticación (`/api/auth/*`).

**Nota**: Antes de implementar estos endpoints, considerar si se necesita:

- Un modelo de base de datos para empleados (tabla `employee` o similar)
- Una tabla opcional para registrar historial de enrolamientos faciales

En la definición del servidor Elysia:

### 3.1 Endpoint de enrolamiento de rostro

- Ruta conceptual: `POST /employees/:id/enroll-face`.
- Mecanismo:
    - Recibir en el cuerpo una representación de imagen (por ejemplo, string base64 sin cabecera de data URL).
    - Buscar el empleado en la DB por ID.
    - Llamar al servicio `enrollFace` con el `employeeId` y los bytes de la imagen.
    - Registrar el resultado si se quiere en una tabla adicional (opcional, puede planearse para otra release).
    - Responder con un objeto tipo `FaceEnrollmentResult`.

### 3.2 Endpoint de identificación por imagen

- Ruta conceptual: `POST /recognition/identify`.
- Mecanismo:
    - Recibir imagen (base64) en el cuerpo.
    - Llamar a `searchFace` del servicio.
    - Si no hay match por encima de un umbral (por ejemplo 80% de similitud), devolver `matched: false`.
    - Si hay match:
        - Recuperar del resultado el `employeeId` (vía `ExternalImageId`).
        - Consultar la DB para obtener el empleado completo.
        - Devolver un objeto `RecognitionResult` que incluya información básica del empleado y el score de similitud.

## Paso 4 – Extender contrato `@sen-checkin/api-contract`

**Estado**: Parcialmente implementado. El paquete `@sen-checkin/api-contract` ya existe y tiene:

- Cliente Eden Treaty configurado (`createApiClient`, `apiClient`)
- Tipo `AppType` exportado que referencia `@sen-checkin/api`
- Configuración base lista para extender

**Tareas pendientes**:

- Extender el tipo `AppType` (y por consecuencia el cliente Eden) con los nuevos endpoints:
    - Método equivalente a `enrollFace(employeeId, imageBase64)` bajo la ruta de empleados.
    - Método equivalente a `identify(imageBase64)` bajo una ruta de reconocimiento.

- Actualizar en pseudocódigo la interfaz del cliente Eden:
    - Definir que desde el frontend se pueda hacer algo del estilo:
        - "cliente.employees.enrollFace(...)" para enrolamiento.
        - "cliente.recognition.identify(...)" para identificación.

- Asegurar que los tipos de entrada y salida usen `@sen-checkin/types`:
    - Entrada: datos de imagen, IDs.
    - Salida: `FaceEnrollmentResult` y `RecognitionResult`.
